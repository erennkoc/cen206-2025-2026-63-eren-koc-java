1. PROJECT DESCRIPTION 
In this term project you will collaboratively design and develop a Java application demonstrating core Object-Oriented 
Programming principles. The same application evolves across two phases: 
 
MIDTERM — Console Application 
FINAL — Swing GUI Extension 
Core Requirements 
• Console-based menu with keyboard navigation 
(arrow / tab)

switchable at runtime from within the application

6. MIDTERM PHASE — Console Application 
6.1 Application Requirements 
• Modular Architecture: Separate library (lib) and application (app) layers. App calls lib. 
• Console UI: Keyboard-navigable menus (arrow keys / Tab). Not number-entry only. 
• Storage Architecture — IRepository Pattern (all 3 required): lib module defines IRepository<T> interface (save, 
findById, findAll, update, delete). Three implementations: BinaryRepository<T> (Serializable + 
ObjectOutputStream/.bin), SqliteRepository<T> (org.xerial:sqlite-jdbc/.db), MySqlRepository<T> (mysql-
connector-j + Docker Compose). RepositoryFactory returns active backend. User switches from Settings screen 
at runtime — no restart required. 
• ER Diagram (all choices): Mandatory regardless of storage choice — PlantUML @startuml entity notation; 
shows entities, attributes, primary/foreign keys, cardinality. Committed as design/plantuml/er.puml + er.png. 
• OOP 4 Principles: Encapsulation, inheritance, polymorphism, abstraction demonstrably applied (LO.1–LO.3). 
• Coverage: 100% JUnit5 unit test coverage and 100% Doxygen documentation coverage.

7. FINAL PHASE — Swing GUI Extension 
7.1 Application Requirements 
• Same Project, New Layer: Extend your console application with a Swing GUI. Do NOT rewrite business logic. 
• MVC Pattern: Strict separation between View (Swing), Controller, and Model (lib). The console layer must still 
work independently. 
• All Console Screens → GUI: Every functional screen you designed in Figma at Midterm must be implemented in 
Swing. 
• Swing Components: Use appropriate components (JFrame, JPanel, JTable, JList, JDialog, JMenuBar, etc.) with 
proper layout managers. 
• Event Handling: All user actions must be handled; input validation and error dialogs required. 
• Design Patterns: Min. 3 patterns from different categories. Observer is strongly recommended for Swing event 
propagation. 
• Code Smell & Refactoring: Identify ≥ 3 code smells; refactor; show before/after in the report. 
• CI/CD: GitHub Actions pipeline: automated build + test on every push; CI badge visible in README. 
• Cross-Platform: Must run on Windows and WSL/Linux.

11. OOP REQUIREMENTS 
11.0 Storage Architecture — IRepository Pattern 
• IRepository<T>: Define in lib module: save(T t), T findById(int id), List<T> findAll(), update(T t), delete(int id). 
Three concrete classes: BinaryRepository<T> (ObjectOutputStream/.bin), SqliteRepository<T> (org.xerial:sqlite-
jdbc/.db), MySqlRepository<T> (mysql-connector-j + Docker Compose). 
• RepositoryFactory: Reads StorageConfig.getActiveBackend() and returns the correct IRepository 
implementation. Config persisted in a .properties file so the choice survives restart. 
• Settings screen: Both console and Swing GUI must expose a Storage Backend menu: [1] Binary  [2] SQLite  [3] 
MySQL. On switch, factory reinitialises — existing data is NOT migrated (state this clearly in the report). 
• Docker: MySQL backend requires docker-compose.yml in repo root. Application shows a clear error message if 
MySQL is unreachable and falls back gracefully. 
• Testing: Each repository implementation must have its own JUnit5 test class. Use in-memory SQLite (:memory:) 
and an H2 MySQL-mode datasource for SQLite/MySQL unit tests to avoid Docker dependency in CI. 
 
11.1 Class Hierarchy 
Design a meaningful class hierarchy. Inheritance must be semantically justified — use composition over inheritance 
when the relationship is not a true 'is-a' relationship. 
11.2 Encapsulation 
• Access modifiers: private, protected, public applied correctly to every member. 
• Getters / Setters: All fields private; expose through accessors.

11.5 SOLID Principles 
• S: Single Responsibility — each class has exactly one reason to change. 
• O: Open/Closed — open for extension, closed for modification.

• D: Dependency Inversion — high-level modules depend on abstractions, not concretes.

17. BONUS FEATURES 
The following advanced features earn bonus points. Core project requirements must be fully met before bonus features 
are considered. Bonus features that are partially implemented or broken will not earn points. 
 
17.1 Database Integration with Docker 
• Database: Replace binary file storage with MySQL or PostgreSQL running in a Docker container. 
• Schema: Define tables, relationships, and constraints in SQL. Provide schema SQL file in repo. 
• Data Access: Use JDBC or JPA/Hibernate with connection pooling. 
• Docker Setup: Provide Dockerfile for the Java app and docker-compose.yml to start app + DB together. 
• Persistence: Configure Docker volumes for database persistence across container restarts. 
 
17.2 Authentication & Authorisation with Keycloak 
CEN206 Object Oriented Programming  |  Term Project Guideline  |  Spring 2025–2026 
RTEÜ Computer Engineering  |  MÜDEK v3.0 Compliant 
27 
• Keycloak: Add Keycloak to your Docker Compose setup. Configure realm, client, and roles. 
• Login/Register: Implement login and registration screens in your Swing GUI via Keycloak. 
• Role-Based Access: Restrict UI features based on user role (e.g. admin vs. user). 
• Session Management: Handle token refresh and logout correctly. 
 
17.3 System Monitoring & Logging 
• Metrics: Integrate Prometheus for metrics collection from your Spring Boot or Java app. 
• Dashboards: Set up Grafana with at least one dashboard showing application health and request rates. 
• Centralised Logging: Use Loki (or ELK) for log aggregation. Use SLF4J + Logback with structured log output. 
• Log Policy: Configure log rotation and define retention policies. 
 
17.4 Microservices with Spring Boot 
• REST API: Develop a Spring Boot REST API that exposes your business logic. Document with Swagger/OpenAPI. 
• Client: Your Swing GUI calls the REST API instead of lib directly. 
• Dockerisation: Each service (API, GUI launcher, DB) in its own Dockerfile; orchestrated via docker-compose. 
• Health Checks: docker-compose depends_on with health checks to control startup order. 
 
ℹ  Bonus Evaluation Criteria 
• Architecture Quality: well-defined layers, separation of concerns. 
• Container Management: correct Docker images, volumes, networking. 
• Security Practices: credentials stored in environment variables — never in source code. 
• Observability: monitoring dashboards and log queries must be demonstrated live. 
• Scalability: explain how your design could be horizontally scaled. 
• Documentation: README covering all advanced setup steps.
